js的microtask 和 macrotask

浏览器的event Loop

浏览器的event Loop至少包含两种 macrotask 和 microtask

Macrotasks包含生成dom对象、解析HTML、执行主线程js代码、更改当前URL还有其他的一些事件如页面加载、输入、网络事件和定时器事件。从浏览器的角度来看，macrotask代表一些离散的独立的工作。当执行完一个task后，浏览器可以继续其他的工作如页面重渲染和垃圾回收。

Microtasks则是完成一些更新应用程序状态的较小任务，如处理promise的回调和DOM的修改，这些任务在浏览器重渲染前执行。Microtask应该以异步的方式尽快执行，其开销比执行一个新的macrotask要小。Microtasks使得我们可以在UI重渲染之前执行某些任务，从而避免了不必要的UI渲染，这些渲染可能导致显示的应用程序状态不一致。

由上图可知

  • macrotask一次只执行一个，而microtask是全部执行完成才开始下一轮。

  • 同一时间只能执行一个任务

  • 任务直到执行完成，不能被其他任务抢断。

当然每一个mircotask 和 macrotask并不一定只有一个队列，也会根据事件的重要程度划分到不同的marcotask和mircotask中，总是优先执行重要的事件。

常见事件的macrotask 和 mircotask区分。

macrotasks: setTimeout ，setInterval， setImmediate，requestAnimationFrame,I/O ，UI渲染
microtasks: Promise, process.nextTick， Object.observe， MutationObserver

由此，我们可以来回答我们的面试题了。

我们都知道，js是单线程的，浏览器执行js的时候是一个macrotask。 遇到setTimeout会将其放入下一个macrotask。
遇到promise我们会将其放入一个microtask。promise。当我们的js全部执行完成以后，我们再去执行microtask,即执行microtask。然后再执行macrotask里面的setTimeout

所以，我们的面试题也就不能得出答案了。。

与性能优化相关。

浏览器通常每秒尝试渲染页面60次，以达到每秒60帧（60 fps），这个帧速率通常被认为是平滑运动的理想选择。这意味着浏览器尝试每16ms渲染一帧。

如果我们的mircotask所有相关的任务大大超过16ms这个时候，浏览器就不能按照正常目标的帧率来渲染ui。用户就能感觉到卡顿。

另外要注意 requestAnimationFrame(code)，一般用于动画，与 setTimeout 方法类似，区别是 setTimeout 是用户指定的，而 requestAnimationFrame 是浏览器刷新频率决定的，一般遵循 W3C 标准，它在浏览器每次刷新页面之前执行。
但有兼容性问题。

结束语

写这篇文章主要是为了下一篇文章，vue异步渲染要用到。属于前置知识。 vue的源码中，用了promise, MumationObserve, setTimeout 根据支持情况，做异步更新。 promise和MumationObserve先于 setTimeout就是因为，mircotask 总是先于ui渲染执行。

MVVM

提到MVVM，好像就不得不说起MVC。 直接上图

我们可以假设，这里有一个数据 M 叫做 name . V中渲染了name的值, C 就是当用户操作V或者C时， C要首先告诉M，
你要做出变化了，M变化完之后，告诉V。我已经变成了另外一个样子了，你要改变了。。

这里就是MVVM。从图上看出，我们只需要把C换成VM就行。用户触发V变化以后，告诉VM, 然后VM告诉M， M变化完成之后告诉C， C在来通知V做相应的变化。。

听上去差别不大，但是，最大的差别就是M和V到底之间到底有没有通信。。。

就以前端为例子
如果页面渲染了一个input, value = name， js里面有一个obj = {name: ‘xiaohong’}, 当用户改变input中的name。obj中的name也随之改变，同时，界面上一个div。里面装着obj.name的值。。

用代码很好实现，

我们可以input的change事件。 当value 变化， 让obj.name = value, 这些完成之后，通知div。你要渲染的值是多少。

这是传统的MVC。

但是，当我们用mvvm时，obj.name改变的时候，我们不再处理告诉div， 你要渲染多少了。只需要告诉VM，我已经变化了，div就是自己发生了变化。。

这就是MVVM。

VUE中的响应式原理

vue.js 主要是采用数据劫持结合发布者-订阅者模式的方式，通过Object.defineProperty()来劫持各个属性的setter，getter，在数据变动时发布消息给订阅者，触发相应的监听回调，实现了数据双向绑定，不再需要我们来维护V，只需要着重维护M如何变化就可以了。

正如上面的代码，我们on 的时候，只是把函数添加进去，emit的时候才是触发函数的时候。这和我们今天要实现的功能有些相似。。 我们then的时候，只是把回调函数传进promise中去，resolve, reject的时候才是真正执行的时候。。

所以我们的代码就可以这么写了

class Prs {
  constructor () {
    this.data = []
  }
  then(fn1,fn2) {
    this.data.push({
      success: fn1,
      fail: fn2
    })
    return this // 实现链式调用。
  }
  resolve(res) {
    if (this.data.length > 0) {
      this.data.shift().success(res)
    }
  }
  reject (res) {
    if (this.data.length > 0) {
      this.data.shift().fail(res)
    }
  }
}

这基本上是promise的最最最最简单，也基本上是最最最核心的原理了，基本上也就是解决了一下 fn1(fn2) 这种写法，让我们能够基本实现链式的写法。当然真正的promise远比这个要复杂很多。我们有时间可以自己慢慢参照规范，慢慢实现。

题外话

这是笔者在日常学习中，对javascript中的this常见的几种场景，做了一个简单的总结。

this 不能在函数创建的时候就确定。只能在函数被调用的时候被确定，并且，this指向的就是函数的执行上下文或者说函数的调用者。

this 的常见的使用情景。

1. 普通函数中的this

上面的代码是在普通函数中的使用，但是实际上， name 这个函数的执行上下文是window || global

name === window.name || global.name

此时函数name的执行的上下文可以理解为window, 或者 global.

所以 this指向的是全局对象。

2. 作为对象的方法。

  • obj.getName()

    这个结果是因为，调用getName 这个函数的调用者是 obj. 所以函数内部的this指向的是obj.

  • var getName = obj.getName

    此时，getName 是一个全局声明的一个变量，这个变量是一个函数，所以，函数的调用者是window || global.

3. 构造函数中的this.

  • xiaoming.getName()
    函数的调用者仍然是xiaoming , 所以函数内部的this 指向的是 xiaoming.

  • xiaomingName()
    函数的调用者变成全局对象, 所以函数内部的this指向的 window || global

4. call, apply, bind

call, apply, bind 这三个方法都改变了函数的执行上下文。 把函数的上下文变成了obj2.

5. 箭头函数

箭头函数不创建新的上下文,没有自己的this. 箭头函数内部的this.就是封闭的上下文中的this。

  • 箭头函数作为对象的方法。

首先上述的写法语法上是没有问题的，但是箭头函数是不会自己绑定this的。所以，这个时候this是空对象。

结论1:箭头函数不应该直接作为函数的方法。
